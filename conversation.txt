  4. Authentication Design

  - Flow: POST /api/auth/login → validate credentials against
  hashed password → issue JWT (HS256, signing key from
  config/user-secrets, short-lived ~60 min) with claims sub,
  username, role.
  - API protection: [Authorize] on task controllers;
  AddAuthentication().AddJwtBearer(...) validates
  issuer/audience/lifetime/signature.
  - Password storage: hashed (BCrypt.Net or ASP.NET
  PasswordHasher). Seed admin gets a pre-hashed password.
  - React protection: token stored client-side, attached via
  Axios interceptor; <ProtectedRoute> guards redirect to /login
  if no/expired token; 401 response → interceptor clears token
  and redirects.

  ▎ Decisions to confirm: (a) token storage — localStorage 
  ▎ (simple, scope-level) vs httpOnly cookie (safer, more
  ▎ setup). I recommend localStorage for this scope. (b) No
  ▎ refresh tokens (out of scope) — re-login on expiry.

● Good — all decisions locked. Let me address the AssignedTo question
  first since it's the real design call, then the seeding/migration
  behavior.

  AssignedTo (string) + AssignedToUserId (FK), Users = auth only

  This is a denormalization decision. Here's the shape I recommend:

  Tasks.AssignedToUserId  int?   FK → Users.Id   (nullable, source of
  truth for "who")
  Tasks.AssignedTo        string?                (display snapshot of the
  name)

  Trade-offs

  Pros
  - No join to render. Task lists/summary show the assignee name straight
  from the row — Users stays purely an auth table, never queried for
  display. This is the main win and matches your "Users for auth only"
  intent.
  - Flexibility. You can assign to a free-text name with AssignedToUserId
  = null (e.g. someone not in the auth table) and link to a real user when
  one exists.
  - Decoupling. Deleting/changing an auth user doesn't break historical
  task display — the snapshot persists.

  Cons
  - Redundancy + staleness. AssignedTo is a copy. If a user's Username
  ever changes, existing tasks keep the old string unless you re-sync. Low
  risk here since auth usernames rarely change, but it's real.
  - Sync responsibility moves to the service. On create/update you must
  decide the rule. I recommend: if AssignedToUserId is provided, the
  service looks up the user and sets AssignedTo from it (FK is source of
  truth); if only a string is given, FK stays null (free-text). One clear
  rule avoids the "which field is correct?" ambiguity.
  - Two fields can disagree if written carelessly — the service rule above
  is what prevents that.

  The alternatives I'm rejecting for you:
  - FK only — needs a Users join every render; contradicts "Users for auth
  only."
  - String only (no FK) — simplest, but zero referential integrity and you
  can't reliably group/report by actual user.

  Keeping both is the right call given your constraint. Confirmed into the
  design.

  Migrations: auto-apply — with one important caveat

  You said "if any entity gets changed, migrations should run
  automatically." Two distinct steps, and only one can be automatic:

  Step: Create a migration from an entity change
  Command: dotnet ef migrations add <Name>
  Automatic?: ❌ No — this is a dev-time, intentional step. There is no
    safe way to auto-generate schema diffs at runtime.
  ────────────────────────────────────────
  Step: Apply pending migrations to the DB
  Command: db.Database.Migrate() at startup
  Automatic?: ✅ Yes — runs on every app start.

  So the workflow is: you change an entity → run dotnet ef migrations add
  X once → and the app applies it automatically next startup (locally and
  in the Docker container). That's the standard and what I'll implement.
  I'm flagging this so there's no surprise that entity edits alone don't
  produce migrations.

  Seeding: Development-only + auto-apply migrations

  Note: EF's HasData (model-level seed) bakes seed rows into migrations,
  so it runs in every environment — that conflicts with "dev only." So
  I'll use a custom idempotent seeder invoked at startup, gated by
  IsDevelopment() instead. Here's the bootstrap (the one piece of code you
  asked for):

  // Program.cs — after building the app, before app.Run()
  using (var scope = app.Services.CreateScope())
  {
      var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();

      // Always apply any pending migrations (local + Docker container)
      await db.Database.MigrateAsync();

      // Seed ONLY in Development, and only if empty (idempotent)
      if (app.Environment.IsDevelopment())
      {
          await DbSeeder.SeedAsync(db);
      }
  }

  // Infrastructure/Seed/DbSeeder.cs
  public static class DbSeeder
  {
      public static async Task SeedAsync(AppDbContext db)
      {
          if (!await db.Users.AnyAsync())
          {
              db.Users.Add(new User
              {
                  Username     = "admin",
                  PasswordHash = /* pre-hashed "Admin@123" via the same
  hasher the app uses */,
                  Role         = "Admin",
                  CreatedAt    = DateTime.UtcNow
              });
              await db.SaveChangesAsync();
          }

          if (!await db.Tasks.AnyAsync())
          {
              db.Tasks.AddRange(/* 10+ sample TaskItem rows, mixed
  Status/Priority,
                                   some with AssignedToUserId+AssignedTo,
  some free-text */);
              await db.SaveChangesAsync();
          }
      }
  }

  Key points:
  - MigrateAsync() runs in all environments → schema always current on
  startup (local LocalDB and the SQL Server container both self-migrate).
  - Seeding is IsDevelopment()-gated → production/staging start clean.
  - Idempotent (AnyAsync() guards) → restarting the app won't duplicate
  rows.
  - Password is hashed with the same hasher the login flow uses, so the
  seeded admin can actually log in.
